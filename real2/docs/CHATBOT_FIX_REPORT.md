# 챗봇 응답 및 데이터 점검 보고서

## 수정한 문제

1. `안녕`, `너 누구야?`, `고마워` 같은 간단한 대화를 축제 검색 실패로 처리하던 문제
2. 모든 질문에 축제 JSON 문맥을 붙여 LLM이 공공데이터에 과하게 의존하던 문제
3. 프런트가 현재 질문만 보내서 `그 축제 요금은?` 같은 후속 질문의 맥락을 잃던 문제
4. 코드 기본 모델과 `.env.example`에 존재하지 않는 `gpt-5.6-luna`가 지정된 문제
5. 추론 토큰이 과도하게 사용되어 실제 답변이 비는 상황을 줄이기 위한 출력 설정 문제

## 수정 후 동작

- 간단한 인사·자기소개·감사·도움말은 축제 검색을 건너뛰고 일반 대화로 처리한다.
- 일반 질문에는 축제 JSON을 LLM 프롬프트에 넣지 않는다.
- 축제명, 지역, 날짜, 무료 여부 등 축제 관련 질문일 때만 JSON 검색 결과를 전달한다.
- 최근 대화 최대 8개를 함께 보내 후속 질문의 맥락을 유지한다.
- 축제 후속 질문은 직전 사용자 질문과 합쳐 다시 검색한다.
- 기본 모델을 `gpt-5-mini`로 통일했다.
- LLM 호출 실패 시 일반 대화와 축제 검색에 맞는 서로 다른 폴백 답변을 사용한다.

## 서울 축제 데이터 확인 결과

- 실제 런타임 import 파일: `netlify/functions/data/seoul-festivals.json`
- 원본 대응 파일: `서울_축제공연행사.json`
- 지역: 서울
- 콘텐츠 유형: 축제공연행사
- 메타데이터 `total`: 201
- 실제 `items` 개수: 201
- 항목별 전체 필드 수: 43
- 원본 대응 파일과 런타임 파일 SHA-256: `8456d828cf582c2765f6a28fee99913f5506d9d461e3146a3ee14261fcf99921`
- 두 파일의 SHA-256이 같으므로 내용은 바이트 단위로 동일하다.

## 데이터 전체를 확인하는 범위

축제 질문이 들어오면 검색 함수는 `items` 201건 전체를 순회한다. 다만 43개 필드를 모두 검색하거나 LLM에 보내지는 않는다.

검색·필터에 사용하는 주요 필드:

- 제목: `title`
- 주소: `addr1`, `addr2`
- 장소: `eventplace`
- 기간: `eventstartdate`, `eventenddate`
- 프로그램: `program`
- 주최·주관: `sponsor1`, `sponsor2`
- 요금: `usetimefestival`
- 연령: `agelimit`

최종 화면과 LLM에 전달하는 주요 필드:

- 제목, 주소, 행사장, 시작일, 종료일, 시간, 요금, 관람 연령
- 전화번호, 대표 이미지, 경도, 위도, 진행 상태
- 프로그램은 최대 240자 발췌

검색 결과는 관련도 순으로 최대 5건만 LLM에 전달한다. 201건 전체와 43개 필드를 매번 모두 전달하면 토큰 비용이 커지고 관련 없는 정보 때문에 답변 정확도가 낮아질 수 있어, 전체 순회 후 관련 후보만 전달하는 구조가 적절하다.
